# プロジェクト横断教訓（xp-rules.md）

Claude.ai の pm-zero プロジェクトの Knowledge にアップロードする。
上限 10 項目。本体 CLAUDE.md に統合されたルールはここから削除する。

## XP-1: Playwright MCP のトークン膨張

- 発生状況: UI 検証で Playwright MCP を使っていた
- 表層の症状: 1 テストで 89-114K トークン消費
- 根本原因: MCP はツールスキーマとスナップショットを毎回コンテキストにインライン展開する
- ルール: UI 検証時 → playwright-slim（73% スキーマ削減）を使う。1 回 10 ステップ以内。検証結果から再利用可能な Playwright テストを生成し、以降は Hook で自動実行

## XP-2: LLM コンテキスト内のタスク状態が失われる

- 発生状況: 長時間セッション、または /compact 後にタスク状態を把握する必要があった
- 表層の症状: 完了済みタスクを未完了と誤認、作成済み API KEY を再作成するよう指示
- 根本原因: LLM のコンテキストウィンドウのみに状態が存在し、コンパクションや注意希薄化で情報喪失
- ルール: タスク状態・決定履歴・エラー履歴 → 外部ファイル（state.md / decisions.md / issues.md）に退避する。Hook で自動維持。セッション開始時に必ず外部ファイルを読む

## XP-3: 環境変数は段階的追記でなく最初から全項目

- 発生状況: プロジェクト初期の環境構築
- 表層の症状: タスク毎に .env.local への追記が発生、何が揃えば完成か不明
- 根本原因: 必要 KEY の全体像を最初に可視化する仕組みがなかった
- ルール: プロジェクト初期化時 → vision.md の Stage 4 で特定した全外部サービスの KEY を .env.example に事前列挙する。各項目に取得先 URL と最新 KEY 名をコメントで記載

## XP-4: 古い学習データに基づく KEY 名・UI 案内の失敗

- 発生状況: 外部サービス（Supabase 等）の KEY 取得手順を案内する時
- 表層の症状: Claude が指示する KEY 名・メニュー位置と実際のサイトが一致しない
- 根本原因: Claude の学習カットオフ後にサービス UI や KEY 名が変更されている
- ルール: 外部サービスの API key 取得を指示する前 → Brave Search で「<サービス名> get API key 2026」「<サービス名> latest KEY name 2026」を必ず検索し、最新情報に基づいて案内する

## XP-5: ユーザー委任は手間とトークンの両方を浪費する

- 発生状況: git push / pnpm install / vercel deploy などを実行する時
- 表層の症状: Claude がユーザーに手動実行を依頼 → ユーザーが別ターミナルで実行 → 結果を貼り付け、というチャットターン増加
- 根本原因: 権限設定が過剰に慎重で、安全なコマンドまで ask になっていた
- ルール: 安全な CLI（git add/commit/push/pull、pnpm、npm、vercel、gh 等）→ settings.json の allow に含め Claude が直接実行する。ユーザーが実行するのはブラウザ操作が必須な行為（GitHub リポジトリ作成、OAuth 承認）のみに限定

## XP-6: Opus 切替で解決率が上がる

- 発生状況: Sonnet で 3 回以上エラー解決に失敗した時
- 表層の症状: 同じアプローチの繰り返しで抜け出せない
- 根本原因: セッション内の仮説バイアスが強化される。モデル変更 + 文脈クリアで思考リセットが必要
- ルール: 3 回目の失敗時 → /escape Skill を起動 → escape.md に試行履歴を固定 → /clear + /model opus で Opus 4.7 に切り替え → 前回試行を明示的に避ける指示で再起動

## XP-7: ブラウザコンソールの全コピペが最速

- 発生状況: サイト上のエラーが 1 回の修正で解決しない時
- 表層の症状: 推測による修正を繰り返し、的外れな変更が積み重なる
- 根本原因: 実際のエラー情報をスタックトレースレベルで取得していない
- ルール: 2 回目の失敗時 → /console-debug Skill を起動 → ユーザーに F12 → Console → 赤いエラー全選択コピー → 貼り付けの 4 ステップを具体的に指示 → 貼り付けられたスタックトレースから原因を特定

## XP-8: 確定的ルールは Hook に置く（CLAUDE.md に書かない）

- 発生状況: prettier 適用・build 確認・型チェック等、100% 遵守が必要なルール
- 表層の症状: CLAUDE.md に書いても Claude がスキップすることがある
- 根本原因: CLAUDE.md は確率的遵守。長大化するとさらに遵守率が下がる
- ルール: 確定的に強制したいルール（prettier / lint / build / type-check / 状態ファイル更新 / エラー記録）→ .claude/hooks/ 配下の Node.js スクリプトとして実装し、適切な Hook イベントに登録する。CLAUDE.md には「判断を伴うルール」のみを残す

## XP-9: コマンド指示は「ターミナル名 + コピペ形式」で明示

- 発生状況: ユーザーが手動実行する必要のある場面
- 表層の症状: 指示を読んで、どのターミナル / シェルで実行すればよいか分からない
- 根本原因: 非エンジニアは PowerShell / bash / WSL の区別を知らない
- ルール: ユーザーに操作を依頼する時 → ① ターミナル名（PowerShell / コマンドプロンプト / Claude Code 内）② コピペ可能な完全なコマンド ③ 期待される出力 の 3 点を必ず明示する

## XP-10: Skill と Hook の責務分離

- 発生状況: 新しい自動化を追加する時にどこに書くか迷う
- 表層の症状: 過去には CLAUDE.md に全部書いていた
- 根本原因: 実装場所のメンタルモデルが曖昧だった
- ルール: 新規ルール追加時の判断基準 → ① 100% 遵守が必要 & 判断不要 = Hook / ② ワークフロー化された手順 = Skill (`.claude/skills/`) / ③ 独立コンテキストが必要な専門処理 = Subagent (`.claude/agents/`) / ④ 判断が必要な毎回のガイドライン = CLAUDE.md
