# PM Zero Claude

**非エンジニアのアイデアを、AI が自律実行できる「再現可能な開発プロセス」に変換するシステム。
使うたびに自分自身が進化し、次のプロジェクトの品質を上げていく。**

---

## 一言で言うと

> 「AI に何を・どう指示するか」を標準化し、誰が使っても同じ品質の結果を出す仕組みを作った。

AI コーディングツール（Claude Code）は、入力の質に出力の質が完全に依存する。
同じツールを使っても、使い方次第で成果に 10 倍の差が生まれる。

PM Zero Claude は、**誰が使っても同じ品質の仕様書セットが出力される仕組み**を設計した。
インタビュー → 5 成果物生成 → 実装 → 教訓抽出 → 次回に反映、というサイクルを自動化している。

---

## 設計した「仕組み」の全体像

```
ユーザーの曖昧なアイデア（自然言語）
        ↓
  構造化インタビュー（4 ステージ / 15〜30 問）
  ─ 誰が話しても同じ品質の仕様書が出るよう設計
        ↓
  成果物パッケージを自動生成（11 ファイル）
  ┌─────────────────────────────────────────┐
  │ 仕様書 / 行動規範 / 権限設定              │
  │ タスク状態管理 / 決定履歴 / エラーログ    │
  │ 自動処理スクリプト（Hooks）               │
  │ 専門エージェント（Subagents）             │
  └─────────────────────────────────────────┘
        ↓
  AI が自律実装（Claude Code）
        ↓
  /ev コマンド → 失敗から教訓を抽出 → ルールとして蓄積
        ↓
  次のプロジェクトの行動規範に自動反映
        ↓
  プロジェクトを重ねるたびに精度が上がる
```

## 3 つの設計上の核心

### 1. プロセスの属人化排除

AI への指示は「書いた人のセンス」に依存する。PM Zero Claude はこれを排除した。

- **構造化インタビュー**：誰がアイデアを出しても、同じ 4 ステージ・同じ質問で仕様書が生成される
- **外部状態管理**：タスクの進捗・API キーの作成履歴・エラーログをファイルに記録し、担当者が変わっても引き継げる
- **暗黙知の可視化**：「なぜこの設計にしたか」を decisions.md に自動記録。後から参照可能

```
❌ 担当者の記憶に依存する進捗管理
✅ state.md / decisions.md / issues.md の 3 ファイルで外部化
   → セッションが終わっても、週が変わっても、状態が消えない
```

### 2. 確率的遵守と確定的強制の分離

AI に「必ずやれ」と文書で指示しても、AI は確率的にしか従わない。
ルールを 2 種類に分類し、それぞれ適切な強制手段を選んだ。

| ルールの種類 | 強制手段 | 遵守率 |
|---|---|---|
| 判断を伴うルール（設計方針・エラー対応） | CLAUDE.md（指示文書） | 確率的 |
| 判断不要な機械的ルール（フォーマット・ビルド・エラー記録） | Hooks（自動実行スクリプト） | **100%** |

ファイル保存のたびに Hooks が自動で `prettier` / 型チェック / エラー記録を実行する。
AI のトークンをゼロ消費で、人間が見落としがちなエラーを確定的に検知できる。

### 3. 失敗から学ぶ自己進化サイクル

プロジェクト完了時に `/ev` コマンドを実行すると、AI がエラーログを分析し、再発防止ルールを自動抽出する。

```
## XP-4: 古い学習データに基づく KEY 名・UI 案内の失敗

- 発生状況: 外部サービスの API キー取得手順を案内する時
- 表層の症状: Claude が指示するメニュー位置と実際のサイトが一致しない
- 根本原因: Claude の学習データ後にサービスの UI が変更されている
- ルール: 外部サービスの設定を指示する前 → 必ず Web 検索で最新 UI を確認する
```

抽出されるルールには原則がある。「このプロジェクト固有の対症療法」は棄却し、
「次のプロジェクトでも通用する抽象化されたルール」のみを採用する。

```
Project 1 完了 → 教訓 2 件蓄積
Project 2 完了 → 教訓 5 件蓄積 → より精度の高い指示書が自動生成される
Project 3 完了 → 教訓 8 件蓄積 → さらに精度が上がる
```

## v6.0 → v7.0 で解決した問題

v7.0 は実際の開発プロジェクト（WEM）で発見した 13 件の構造欠陥を根本解決した。
特に深刻だったのは「**完了済みタスクを AI が未完了と誤認する**」問題だった。

### 発見した根本原因

```
❌ v6.0 の構造：
   タスクの状態 → AI の「記憶（コンテキスト）」の中にのみ存在
   長時間開発が続く → コンテキストが圧縮される → 状態情報が消える
   → 「このタスクは完了しましたか？」と聞くと AI が誤認して答える

✅ v7.0 の解決策：
   タスクの状態 → 外部ファイル（state.md）に記録・Hook が自動更新
   コンテキストが圧縮されても → ファイルは消えない → 状態は維持される
```

### v7.0 の 4 つのパラダイムシフト

| 変更 | v6.0 | v7.0 |
|---|---|---|
| 状態管理 | AI の記憶（揮発性） | 外部ファイル 4 本（永続） |
| ルール強制 | 指示文書（確率的） | Hook スクリプト（確定的） |
| 役割分担 | AI 1 体がすべて担当 | 専門 AI 3 体が分業 |
| 手順の再利用 | セッションごとにゼロから | Skill として保存・呼出可能 |

## 定量的効果

| 改善点 | 効果 |
|---|---|
| コンテキストの 90% がキャッシュ読取に | トークンコスト 70〜80% 削減 |
| インタビューバッチ設計（3〜5 問 / ターン） | インタビューコスト 75% 削減 |
| Playwright → playwright-slim 切替 | 検証トークン 73% 削減 |
| Hooks による確定的テスト自動化 | エラーの 80% をゼロトークンで検知 |
| 専門 AI（Subagent）分業による汚染分離 | コンテキスト圧縮頻度の低下 |
| **合計** | **v6.0 比 40〜60% コスト削減** |

## 使い方

### 1. セットアップ（1 回のみ）

Claude.ai でプロジェクトを作成し、2 ファイルをナレッジにアップロードする。

```
ナレッジ ← pm-zero-claude-knowledge-v7.0.md
ナレッジ ← xp-rules.md
```

グローバル MCP を登録する（Claude Code のターミナルで実行）。

```bash
# 公式ドキュメント参照 MCP（API 仕様の推測実装を防止）
claude mcp add context7 -s user -- npx -y @upstash/context7-mcp@latest

# Web 検索 MCP（エラー解決・外部サービスの最新 UI 確認）
claude mcp add brave-search -s user -e BRAVE_API_KEY=YOUR_KEY -- npx -y @brave/brave-search-mcp-server
```

### 2. アイデアを話す

Claude.ai のプロジェクトに作りたいものを 1〜2 文で伝える。
PM エージェントが 4 ステージ（目的→範囲→振る舞い→制約）で 15〜30 問インタビューし、成果物を生成する。

### 3. 実装する

生成された成果物をプロジェクトフォルダに配置し、Claude Code を起動する。

```bash
claude
# 起動後に入力：
# @vision.md を読んで、Task Breakdown の最初のタスクを実装してください
```

### 4. 教訓を蓄積する（プロジェクト完了時）

```bash
/ev
# エラーログを分析 → 再発防止ルールを抽出 → xp-rules.md 用に出力
# xp-rules.md を Claude.ai に再アップロードで次回から自動反映
```

---

## 特殊コマンド

| コマンド | タイミング | 処理内容 |
|---|---|---|
| `/ev` | プロジェクト完了時 | 教訓抽出 → xp-rules.md 更新 + README.md 生成 |
| `/resume` | 長期中断後の再開時 | 状態ファイルと実コードを照合し現状を報告 |
| `/console-debug` | ブラウザエラーが解決しない時 | コンソール出力の取得方法を案内し原因を特定 |
| `/escape` | エラーが 3 回解決しない時 | 試行履歴を記録 → 上位モデルへエスカレーション |
| `/audit` | 月 1 回推奨 | ナレッジの陳腐化チェックと更新案の提示 |
